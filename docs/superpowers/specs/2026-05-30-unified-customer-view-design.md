# WEDLY 통합 고객 화면(고객 360) 설계

> 작성일: 2026-05-30
> 관련 문서: `2026-05-30-shared-detail-modal-shell-design.md` (공통 상세창 모듈화 — 이 화면의 "자세히 보기" 부품)
> 구현 대상 앱: **wedly-erp** (본부 앱). 파트너 앱(하이브·일루아·향후 추가)은 같은 구조의 "범위만 다른 창문".

---

## 1. 배경 — 지금 구조로는 왜 안 되나 (조사 근거)

한 고객(한 회사)이 여러 업무 영역에서 동시에 일이 진행된다:
경정청구 · 정책자금 · 무상지원금 · 노무지원금 · 기업인증 · 특허 등.
영역마다 담당 파트너사가 다르다(예: 경정=하이브, 정부지원금=일루아).
"하이브에 입력된 고객의 정부지원금을 일루아가 담당"하는 교차 상황도 있다.

**목표**: 하나의 시스템에서 고객 한 명을 열면, 그 고객의 **모든 영역 진행상황을 영역별로 구분해서** 보고, **관련 자료도 전부** 확인할 수 있어야 한다. 입력은 간편하고, 확인은 직관적이어야 한다.

**현재 상태(코드 조사로 확인한 사실):**
- 영역마다 **별도 저장소**가 따로 있다 — 경정청구·정책자금·무상지원금·노무지원금이 서로 참조 없는 독립 목록(각각 JSON 형태 저장 + 노션 동기화 일부).
- 같은 고객이 여러 영역에 걸쳐도 **각 목록에 남남처럼 따로** 적혀 있다. 줄과 줄을 잇는 끈이 없다.
- 상호·대표·연락처·사업자번호·통합관리코드 같은 **공통 식별 정보는 각 목록이 자기 안에 중복 저장**한다.
- 연락처를 정규화해 대조하는 **통합 목록이 절반쯤 있으나**, 중심 뼈대로 안 쓰이고 연락처는 바뀌어 헐겁다.
- 파트너 담당은 영역마다 제각각(경정=위들리/하이브 선택칸, 정책=애드스파크/JJK/WEDLY 선택칸).
- 앱 관계: 하이브는 ERP의 경정청구 저장소를 "하이브 담당분만" 골라 읽음. 하이브는 정부지원금 저장소(별도)에도 연결. ERP가 영역 저장소들의 주인.

→ 이번 모듈화(공통 상세창)는 "각 영역 상세창의 생김새·동작을 통일"하는 바탕 공사다. 꼭 필요하지만, **"고객을 가로질러 묶어 보여주기"와는 별개**라서, 모듈화만으로는 위 목표가 채워지지 않는다. (사용자가 느낀 그대로 — 정확한 진단.)

## 2. 핵심 개념 모델

지금의 중심 단위는 "영역의 한 줄"이다. 그 위에 **"고객 한 명"**이라는 상위 칸을 둔다.

```
고객 (회사 1곳, 사업자등록번호 기준)
├─ 업무: 경정청구   (담당 파트너 · 진행상태 · 자료)
├─ 업무: 정책자금   (담당 파트너 · 진행상태 · 자료)
├─ 업무: 무상지원금 (…)
├─ 업무: 노무       (…)
├─ 업무: 기업인증   (아직 영역 없음 — 자리만)
└─ 업무: 특허       (아직 영역 없음 — 자리만)
```

- **고객(Customer)**: 회사 한 곳. 기본정보(상호·대표·연락처·사업자번호 등)는 여기 한 번만.
- **업무(영역 건)**: "고객 × 영역" 한 건. 업무마다 담당 파트너 + 진행상태 + 자료가 붙는다.
- **파트너사(Partner)**: 코드에 박지 않고 **등록 명단**으로 관리. 새 파트너는 등록만으로 추가.
- **영역 섹션(Domain Section)**: 화면에서 한 영역을 나타내는 단위. 어떤 저장소를 어떻게 연결할지 **설정**으로 정의(§7).

### 2-1. 고객을 묶는 열쇠 = 사업자등록번호 (확정)
- 가장 안 변하는 공식 번호. 영역 저장소마다 사업자번호가 들어 있어 묶기 기준으로 적합.
- **예외 처리**: 사업자번호가 아직 없는 초기 가망고객은 **연락처로 임시 연결**했다가, 사업자번호가 생기면 자동으로 이어붙인다.
- 묶기는 **기존 저장소를 갈아엎지 않고**, 그 위에서 사업자번호로 같은 고객을 모으는 "묶는 층"으로 구현(안전).

## 3. 화면 구조 (UI) — 두 단계 탭

지금 쓰는 **상세 모달 구성을 그대로 두고, 그 위에 "영역 고르는 층"을 한 겹 얹는다.** 두 단계 모두 가로 탭.

```
1단계 (영역 탭):  [기본정보*] [경정청구] [정책자금] [무상] [노무] [인증] [특허]
                      │
                      └─ 예: [경정청구] 클릭 시 ↓
2단계 (세부 섹션): [히스토리*] [미팅정보] [계약정보] [정산정보] [환불정보] [파일]
                      └─ 선택한 세부 섹션 내용 표시

(* = 고정. 나머지는 드래그로 자리 바꾸기 가능)
```

### 3-1. 1단계 — 영역 탭
- 맨 앞 **`기본정보`는 고정(맨 왼쪽)**. 누르면 고객 기본정보(상호·대표·사업자번호·연락처 등).
- 나머지 영역 탭(경정·정책·무상·노무·인증·특허)은 **드래그로 자리 바꾸기 가능**.
- 각 탭에 **작은 상태 표시(점·색)** → 한 영역만 펼쳐 봐도 어느 영역이 진행 중인지 탭 줄에서 보임.
- (선택) **`기본정보` 탭 안에 전체 영역 현황 한 줄 요약**을 같이 두어 "한눈에 보기"를 살린다.
- 영역이 아주 많아지면 탭 줄이 옆으로 밀리거나 "더보기"로 접힌다.

### 3-2. 2단계 — 세부 섹션 (= 지금 상세 모달 구성 그대로)
- 영역 탭을 누르면, **지금 상세 모달에 있는 그 세부 섹션들**이 나온다.
- 맨 앞 **`히스토리`는 고정(맨 왼쪽)**. 나머지(미팅정보·계약정보·정산정보·환불정보·파일 등)는 **드래그로 자리 바꾸기 가능**.
- **영역마다 세부 섹션이 다를 수 있다** — 경정청구는 미팅·계약·정산·환불·히스토리·파일, 정책자금·무상은 그 영역 설정에 맞는 세부 섹션만. (각 영역의 설정대로 — 모듈화의 핵심)
- **안쪽 상세 모달 구성은 손대지 않는다.** 지금 상세 모달이 "한 영역의 2단계"가 되고, 그 앞에 "기본정보+영역"이라는 1단계가 새로 생기는 것뿐.

### 3-3. 끼워넣기형(팝업 아님)
- 영역 탭 아래 세부 내용은 **팝업이 아니라 그 자리에 끼워서** 보여준다(팝업 위 팝업 방지).
- 단, 파트너 앱(하이브·일루아)에서 목록의 한 줄을 클릭하는 기존 동선은 **팝업으로** 유지.
- → 같은 상세창 내용물을 **"팝업으로도, 끼워넣기로도"** 띄울 수 있어야 한다. ("내용물"과 "팝업 테두리(닫기·뒤 가림막)" 분리. → §8, 모듈화 작업에 요건 추가.)

### 3-4. 순서 저장 범위 (드래그로 바꾼 위치)
- **기본 방침: 공유 설정**(모두가 같은 순서를 본다). 지금 상세 모달의 항목/섹션 순서가 서버에 저장되어 공유되는 방식과 동일하게 맞춘다.
- (열린 결정 — §11에 기록. 개인별 순서가 필요하면 그때 확장.)

## 4. 앱별 "창문"(범위) + 설정형 노출

정보(고객·업무)는 **한 곳에 한 벌**만 두고, 앱마다 그 정보를 **다른 범위로 보는 창문**이 된다.

- **ERP 창문**: 모든 파트너·모든 영역을 다 봄 → 여기에 통합 고객 화면(§3)이 들어감.
- **파트너 앱 창문(하이브·일루아·향후 추가)**: 구조는 같고, **"내 담당 업무만"** 골라 보여줌(지금 하이브가 경정청구에서 하이브 담당분만 보는 방식을 전 영역으로 일반화).
- **새 파트너**: 창문 하나 더 = 코드 거의 그대로. (파트너 등록 + 그 파트너용 범위 설정 + 배포)

### 4-1. 파트너 간 노출 = 설정형 (확정)
- 기본은 "내 담당 업무만".
- ERP에 **"이 고객(또는 이 업무)을 어느 파트너에게 보일지" 켜는 공유 스위치**를 둔다. 켜진 것만 상대 파트너 창문에 나타남.
- 교차 상황(하이브 입력 고객의 정부지원금을 일루아가 담당)이 업무 단위 담당 + 노출 스위치로 자연히 표현됨.

---

## 5. 입력 간편화

- "새 업무 추가" → ① 고객 검색(사업자번호·상호) → 있으면 선택(기본정보 자동 채움) / 없으면 새 고객 한 번 등록 → ② 영역 선택 → ③ 담당 파트너 선택 → 끝.
- **같은 고객 정보를 영역마다 다시 타이핑하지 않는다.**
- 새 고객 등록 시 사업자번호로 **중복 점검**(이미 다른 영역에 있는 고객이면 같은 고객으로 묶음).

---

## 6. 안전 원칙

- 기존 영역 저장소·노션 동기화는 **그대로 둔다.** 그 위에 "고객으로 묶는 층 + 통합 화면"만 얹는다.
- 한 영역·한 파트너로 먼저 시범 → 검증 → 확장.
- 프로덕션 저장소 읽기/변경은 그때그때 명시 승인 후 진행.

---

## 7. 영역 섹션 연결 설정 (확장성 핵심)

새 영역 섹션을 추가하거나 기존 섹션을 손볼 때, **화면에서 연결을 설정**한다. 막연한 "아무거나 연결"이 되지 않도록 고르는 항목을 4갈래로 나눈다.

| 항목 | 내용 |
|---|---|
| **① 데이터 출처** | 어떤 앱 · 어떤 페이지 · 어떤 저장소에서 가져올지 |
| **② 고객 연결 매핑** | 그 출처에서 *사업자번호·상호·대표·연락처*가 각각 **어느 칸**인지 지정. 이게 있어야 그 줄이 고객과 묶임. (출처마다 칸 이름이 달라 필수) |
| **③ 표시·탭 구성** | 이 섹션에서 어떤 항목을 어떤 순서로, 어떤 세부 섹션(미팅/계약/정산/환불/히스토리/파일)을 켤지 |
| **④ 노출 대상** | 어떤 앱·파트너가 이 섹션을 보는지 + "내 담당만 거르기" + 설정형 노출(§4-1) |

①만 있으면 "데이터는 보이는데 어느 고객 건지 못 묶는" 문제 → **②(연결 매핑)가 짝으로 필수.**

**범위는 단계적으로(중요):**
- **먼저**: 지금 있는 출처들(경정·정책·무상·노무)을 "고를 수 있는 목록"으로 미리 등록해 두고, 설정 화면에서 "어느 섹션 ↔ 어느 출처"를 고르고 ②③④를 채우는 수준. (개발자는 출처 등록만, 운영자는 화면에서 연결. 가볍고 안전.)
- **나중에**: 완전히 새로운 외부 저장소도 화면에서 직접 등록·연결. 필요해질 때 확장. (사실상 작은 연결 플랫폼 — 처음부터 만들면 위험·비용 큼.)

## 8. 기존 모듈화(공통 상세창)와의 관계

- 통합 고객 화면의 **2단계 세부 섹션 = 모듈화 중인 공통 상세창의 내용물**. 버리는 일이 아니라 이 큰 그림의 핵심 부품.
- **모듈화 작업(특히 진행 중인 본체 일반화 단계)에 요건 추가**: 상세창을 **팝업으로도, 화면 안에 끼워서도** 띄울 수 있게 — "내용물"과 "팝업 테두리"를 분리.
- 드래그로 자리 바꾸기: 지금 상세창의 "항목 끌어 옮기기" 방식을 **탭/섹션 순서 바꾸기**로 넓혀 재사용.

---

## 9. 단계 나누기 (각 단계가 독립적으로 동작·검증 가능)

| 단계 | 내용 | 위험 |
|---|---|---|
| **0 (시범 — 시작점)** | ERP에서 기존 영역 정보를 **사업자번호로 묶어 "고객 통합 보기"를 읽기 전용으로**. 기존 정보 안 건드림. "한 고객 한눈에"를 가장 빨리·안전하게 증명 | 매우 낮음 |
| **1** | 통합 화면에서 **두 단계 탭 + 끼워넣기형 상세** 연결(모듈화 결과 재활용). 1·2단계 드래그 순서(기본정보·히스토리 고정) | 낮음 |
| **2** | **파트너 등록 명단 + 업무별 담당 + 창문별 추려보기 + 설정형 노출** | 중간 |
| **3** | **간편 입력**(고객 먼저 → 자동 채움 → 영역·담당 선택) + 새 고객 중복 점검 | 중간 |
| **4 (나중)** | **기업인증·특허 영역 신설** + 영역 섹션 연결 설정 화면(§7 먼저 단계) | 중간 |

## 10. 위험 / 본질적 한계

- 사업자번호 없는 초기 가망고객 묶기는 연락처 임시 연결 → 오묶임 가능성(중복 점검·수동 보정 필요).
- 출처마다 칸 이름·형식이 달라 ②연결 매핑을 정확히 채워야 함(빠지면 못 묶음).
- 외부 저장소 자유 연결(§7 나중 단계)은 본질적으로 큰 작업 — 단계적으로만.
- 공용 패키지 단독 타입검사 불가 → 모듈화 부분 최종 검증은 하이브 빌드 시점.

---

## 11. 열린 결정 (구현 계획 전 확정 필요)

1. **드래그 순서 저장 범위**: 공유(모두 같은 순서) vs 개인(각자 정렬). — 기본안: 공유(현 방식과 동일).
2. **`기본정보` 탭의 "전체 현황 요약"** 포함 여부. — 기본안: 포함(한눈에 보기 보존).
3. **외부 저장소 자유 연결(§7 나중 단계)** 착수 시점. — 기본안: 0~3단계 검증 후로 미룸.
4. **파트너별 특수 화면** 필요 시 별도 개발 범위(대부분은 설정·범위로 해결).

## 12. 확정 요약

- 고객 묶는 열쇠 = **사업자등록번호**(가망은 연락처 임시).
- ERP = 통합 본부 / 파트너 앱 = 범위만 다른 창문 / 파트너는 등록으로 확장.
- 파트너 간 노출 = **설정형**(기본 분리, ERP에서 필요한 것만 켬).
- 화면 = **두 단계 가로 탭** (1단계 기본정보 고정+영역 드래그 / 2단계 히스토리 고정+세부 드래그), **끼워넣기형 상세**.
- 영역 섹션은 **연결 설정**(출처·매핑·표시·노출)으로 추가/수정, **단계적** 확장.
- 공통 상세창(모듈화)은 2단계 세부 섹션 부품으로 재활용 + "끼워넣기 가능" 요건 추가.
- 시작 = **0단계**(읽기 전용 고객 통합 보기).
