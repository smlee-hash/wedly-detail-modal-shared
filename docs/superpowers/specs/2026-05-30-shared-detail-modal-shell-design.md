# 세 앱 상세창 "공용 틀 하나로 통합" 설계

작성일: 2026-05-30 · 대상: 하이브 / 일루아 / ERP(5개 모듈) · 공용 부품: `@wedly/detail-modal-shared`

## 1. 목표 (사장님 지시)

세 앱(하이브·일루아·ERP)의 상세창을 **하이브 구조의 공용 틀 하나**로 통일한다.
앱마다 다른 것은 다음뿐이어야 한다:

1. **컬럼 이름** (필드 라벨·키)
2. **컬럼 순서·위치** (어느 섹션/탭에 어떤 순서로)
3. **켤 탭 선택** (예: 일루아는 회의 안 켬)
4. **앱별 서버 연결** (댓글·이력·데이터 저장 위치 — 앱마다 서버가 다름)

"틀(겉 모양·탭 배치·동작)"은 100% 공용, 위 1~4만 앱별 설정.

## 2. 현재 상태 (실제 코드 확인 결과 — 추측 아님)

- **공유되는 건 탭 조각뿐**: 정산·회의·파일 탭은 공용 부품을 같이 쓴다. **상세창 전체 틀은 공유 안 됨.**
- **손으로 짠 틀 파일이 7개**:
  - 하이브 1개 (3,225줄) — 이미 `sections`·`customPanels`·다수 콜백을 *넘겨받는 값*으로 처리하는 **가장 진화된 거의-공용 형태**
  - 일루아 1개 (1,688줄) — **회의 탭 없음**, 댓글은 이력 안에 인라인
  - ERP 5개 (904~2,097줄) — 서로 **복붙 수준**, 탭이 한 줄에 평면 배치
- **공용 폴더에 미사용 초안 1개** (2,598줄) = **하이브 상세창을 복사해 둔 것** (경정 키는 하이브 자체 도메인 키). 일반화·내보내기(export) 안 돼 아무도 안 씀.
- **의존 방식**: 세 앱 모두 `github:smlee-hash/wedly-detail-modal-shared`. → 공용 부품을 고쳐 올리면, 각 앱이 **다시 설치할 때** 반영(자동 아님).

## 3. 핵심 통찰

하이브 상세창은 **이미 거의 공용 틀**이다(섹션 목록·패널·여러 동작을 입력값으로 받음). 아직 하이브 안에 박혀 있어 떼어내야 할 것은 **딱 4가지**:

| 박혀 있는 것 | 현재 | 공용화 방법 |
|---|---|---|
| ① 컬럼 정의 | `CONTRACT_FIELDS` 상수가 코드에 박힘 | **넘겨받는 값(prop)** 으로 |
| ② 권한 | `useAccess()` 직접 호출(하이브 전용) | `isAdmin` **넘겨받는 값** 으로 |
| ③ 서버 주소 | `/api/entries/...`, `/api/detail-section-mapping/...` 등 고정 | **데이터 연결 어댑터** 로 주입 |
| ④ 댓글·이력 | `HistoryPanel`(하이브 전용) + 고정 댓글 주소 | **이력 연결 어댑터**(불러오기/쓰기/삭제 + 카테고리) 로 주입 |

이 4가지만 "앱이 넘겨주는 값"으로 바꾸면 하이브 틀이 그대로 공용 틀이 된다.

## 4. 설계 — 공용 틀이 받는 "설정 표면"

공용 `DetailModal`(새로 export)이 받는 값:

1. `fields` — 컬럼 정의(이름·타입·포맷·키). **[앱별]**
2. `sections` — 섹션 목록 + 순서 + `fieldKeys`(컬럼 위치) + `kind`(fields/settlement/meetings/files/tiered-*). **[앱별]** *(이미 존재)*
3. `isAdmin` — 권한. **[주입]**
4. `dataSource` — 데이터 연결 어댑터: row 저장(patch)·신규 생성·파일 업로드/다운로드 경로·차수필드 읽기/쓰기·섹션매핑 읽기/쓰기. **[앱별]**
5. `historySource` — 이력·댓글 연결 어댑터: 불러오기/등록/삭제 콜백 + 카테고리 정의. (또는 이력 패널을 통째로 넘기는 방식) **[앱별]**
6. 탭 렌더링은 `kind` 별로 이미 공용 탭(정산·회의·파일) 사용.

→ **틀·탭 배치·동작은 100% 공용**, 위 1~5만 앱별. ERP 5개 모듈의 복붙 틀(합계 7천여 줄)이 **작은 설정 파일 5개 + 공용 틀 1개**로 줄어드는 게 진짜 이득.

## 5. 안전 계약 (절대 안 깨뜨릴 것)

- **정산(돈) 탭**: 이미 공용 — 저장 방식·계산 **그대로**.
- **파일 저장**: 칸별 저장 방식 **그대로**(기존 `ErpFilesPanel` 어댑터 유지) → 일루아 동기화·외부 연동·이력·보정 무영향.
- **댓글·이력**: 앱별 서버에 **그대로 저장**(노션 / DB / REST). 틀만 공용, 연결은 앱별 어댑터로.
- **데이터 읽기/쓰기 경로**: 각 앱 기존 경로 **변경 없음** — 어댑터로 그대로 호출.
- 하이브·일루아·ERP **공용 부품 코드는 신중히** — 한 곳 고치면 세 앱에 영향. 단계마다 빌드·검증.

## 6. 단계적 적용 (안전 최우선 — 한 단계씩 검증 후 다음)

- **0단계**: 이 설계서 검토·승인.
- **1단계 (시범 = 하이브 자기 자신)**: 하이브 상세창을 공용 폴더로 옮겨 일반화·export. 하이브가 그 공용 틀을 가져다 쓰되 *자기 설정*을 넘김. → **하이브 화면 변화 0**(자기 코드라 동작 동일) = 추출 성공 증명. **가장 안전한 첫 단계.** 미사용 초안(2,598줄)은 이 일반화 버전으로 교체(삭제).
- **2단계 (ERP 1개 모듈 시범)**: 댓글이 가장 단순한 모듈 1개(**free-subsidy**, REST 댓글, 904줄)를 공용 틀로 교체. 탭 전환·저장·댓글·파일 검증.
- **3단계**: 나머지 ERP 4개(government·policy·tax·labor) + 일루아 순차 교체. 일루아는 회의 안 켬.
- 각 단계: 빌드 → 테스트 → 화면 동작 점검 → 커밋 → 푸시 → (해당 시) 공용 부품 새 버전 → 앱 재설치 → 배포 SUCCESS 확인 → PR. **회귀 시 직전 단계로 복원.**

## 7. 위험·결정사항

- **ERP 탭 배치 변경**: 평면 배치 → 하이브식(상세정보 안에 접어 넣기). 보이는 모습 변함. **사장님 승인 완료.**
- **댓글·이력 서버 차이**(노션 polling / `_comments` DB / REST): 어댑터로 흡수 — **가장 손 많이 가고 위험한 부분.** 2·3단계에서 모듈별로 신중히.
- **공용 부품 = github 의존**: 한 곳 고쳐 올려도 각 앱이 다시 설치해야 반영(자동 아님). 단계마다 재설치 확인.
- **일루아 차이**(회의 없음, 이력 카테고리 4종): "켤 탭 선택" + 카테고리 설정으로 흡수.

## 8. YAGNI (안 할 것)

- 모든 앱 댓글을 한 서버로 통합 — **안 함**(위험·범위 밖). 틀만 통일, 저장 위치는 앱별 유지.
- 미사용 초안 2,598줄 살리기 — 1단계 일반화 버전으로 **교체(삭제)**.
- 도메인 로직(정산 계산·환불 차수·파일 분류) 손대기 — **안 함**. 그대로 둠.

## 9. 검증 방법

- 단위 테스트: 파일 어댑터(이미 64개) + 새 "데이터/이력 연결 어댑터" 변환 로직.
- 단계별 빌드 + 화면 동작 점검(탭 전환·필드 저장·댓글 등록·파일 업로드/삭제).
- 1단계는 "하이브 동작 0 변화"가 합격 기준.
